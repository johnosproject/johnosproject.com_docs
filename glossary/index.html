Glossary

The John Project​

John O.S. Project

It’s the Open Source project that provides all softwares and tools to create a complete and easy to use IoT EcoSystem.

This project contains all source code required to build the components (John Object Daemon, John Service Library and the John Cloud Platform) required by any other IoT (John) Solution.

John O.S. Protocol

It’s the language used by John Objects and John Services to share object’s info and interact with them, both for Direct Communication and Cloud Communication.

John Object Daemon

The software that represents an John Object in a John EcoSystem.

His main tasks are to share the Object’s info to the John EcoSystem and translate the John O.S. Protocol to/from John Object’s Utils & Apps. Once a John Object is shared on the John EcoSystem, John Services can interact with it.

The JOD can be configured to adapt it to any kind of object. It’s main configuration is the John Object Structure that hierarchically defines which Status and Actions expose the Object to the John EcoSystem. Status and Actions can handle JOD requests in different ways depending on the Utils & Apps availables for the JOD instance.

Makers can package (1) JOD, (2) his configs, (3) required Utils & Apps and create a John Object Distribution ready to distribute and to install.

In a John EcoSystem, a John Object is identified by his ID and his model.

Same John Object’s models share the same John Object Structure.

John Service Library

Multi-language library that can be included by developers in their softwares.

Once the JSL is included in a software then it can be considered a John Service.

The JSL helps developers to list, interact and manage user’s objects. It also provides John User’s authentication methods.

John Cloud Platform

The (optional) cloud component for a John EcoSystem.

The JCP is a microservice cloud platform that can be executed locally (for development purpose) or deployed on cloud.

Public John Cloud Platform

The shared John EcoSystem. It’s a public accessibile John Cloud Platform instance reachable at www.johnosproject.org/frontend/index.html.

Any John Object connected to this JCP can be accessible from anyone (after object’s permissions check); the same for all John Services configured to connect to Public JCP.

The IoT EcoSystem​

IoT EcoSystem

The set of all components and their configs required by an IoT Solution.

Normally there are some physical components (the Object) and some user’s interfaces or batch/storage processes (the Service).

Between them can be dozens of technologies: electronic components, hw interfaces, communication modules, gateways, cloud services, authorization servers…

Finally the IoT EcoSystem’s scalability depends on the number and size of the IoT Solutions deployed. The IoT EcoSystem must be scalable in step with the growth of deployed IoT Solutions.

IoT Solution

Every time an Object communicates with a Service, we can see it as an IoT Solution.

IoT Solutions are based on only 3 components: Objects, Services and optionally a cloud service for remote communication. Then these components can be used by 3 actors: Makers, Developers and End Users.

With this set of components and actors, you can design any type of IoT Solution like Data Collection, Digital Twin, Predictive Maintenance, Car Sharing and many many more.

Users​

Makers

The IoT EcoSystem actor that designed the Object.

Maker is anyone that creates a connected Object. He can be a hobbyist or a manufacturer that produces and sells connected Objects.

Developers

The IoT EcoSystem actor that implemented the Service.

Any developer that must interact with a connected Object. Typically, he must integrate different standards and approaches, depending on how many technologies are implied in the solution developed. He is also in charge of implementing object’s interactions logic and user interfaces.

End Users

The IoT EcoSystem actor that uses the Object and/or Service.

Users, in an IoT EcoSystem, can interact with an Object directly (physically) or remotely (via a Service). Normally, Services are also used by Users to configure and manage their Objects.

Components​

Object

Any connected Object, regardless of the technologies used for his communication. But depending on that, it can be used only on certain IoT Solutions.

Object’s Firmware

Each Object is composed of HW and SW components.

The Firmware, typically it’s the low-level SW dedicated to control the HW components (read sensor data and/or activate actuators).

Firmware is used by other SW components that Tx/Rx sensors data and actuators commands to other IoT EcoSystem components (generally to aggregator or storage components; local or cloud).

Service

Any software that elaborate/store Object’s data and/or interact with it.

Typically, Services are User Interfaces like Mobile Apps or Web Dashboards. Other times Services can be AI algorithms or Data Lakes.

Infrastructure softwares, like agents, gateways, load balancers, APIs…; are not considered Services of an IoT EcoSystem. because they are part of the IoT EcoSystem’s infrastructure.

Communication​

Direct Communication

When a connected Object and a Service communicate directly, without intermediates like cloud platforms or gateways.

Cloud Communication

When a connected Object and a Service communicate through intermediates like cloud platforms or gateways.

The John EcoSystem​

John EcoSystem

The John EcoSystem is an IoT EcoSystem based on John O.S. Project’s components.

The John EcoSystem allows Makers create John Objects in minutes without coding skills. At the same time Developers can include the John Service Library in their software to make them John Services.

A John EcoSystem can be self hosted for development or security reasons, or can be used the Public John Cloud Platform as a shared John EcoSystem.

Users​

John User

It’s an End User of a John EcoSystem.

Components​

Object​

John Object

The Object of a John EcoSystem.

Every running instance of the John Object Daemon represents a John Object.

A John Object can be bought in a shop; or a user can download and install a John Object Distribution; or even a Maker can configure a JOD instance and build his own John Object.

Each John Object is configured with a John Object Structure that defines his Status and Actions. Then, End Users can query Object’s Status and/or execute his Actions from John Services.

Depending on the technologies used in the connected Object, John Objects are splitted in 3 groups: John Native Objects, John Gateway Objects, John Web Objects.

John Native Object

An object is considered a John Native Object when his States and Actions interact directly with the object's HW.

This kind of John Objects are composed of HW (processor, local storage, communication module, sensor and/or actuators) and SW (JOD, Utils & Apps and optionally extra firmware) components. For example a SoC board like Raspberry Pi in conjunction with some sensors and actuators, can be used as HW for a John Native Object.

John Gateway Object

A John Object can also represent a set of objects, for example all devices managed by a home automation gateway/hub.

Many home automation, but also industrial, standards use local gateways to aggregate multiple devices communication. Sometimes those local gateways act as bridges from device’s standard and IP networks and (optionally) can expose a web server as UI or API interfaces.

When configured to represent this kind of objects, the JOD can be installed directly on the local gateway (that requires sysadmin access to the gateway, that invalidates the local gateway’s warranty). If that’s not possible, the JOD can be also installed on another local machine that can access the local gateway’s UI or API.

John Web Object

When a connected Object exposes its info and features from a cloud service, we can integrate it to a John EcoSystem, as a John Web Object.

In a John Web Object his States and Actions send HTTP requests to query/execute the object's status/actions.

John Object Distribution

A pre-configured and ready-to-run John Object Daemon distribution.

John Object’s Utils & Apps (Utils & Apps)

Utils & Apps extends the John Object Daemon and acts as Object’s Firmware.

That mean, the JOD interact with the real world (outside the John EcoSystem) via Utils & Apps calls.

Depending on configured Utils & Apps in the John Object Structures, Makers can create John Objects that interact with HW modules (GPIO, I2C, 1-wire..), other protocols (MQTT, MBus, KNX…) or even with web pages (HTTP APIs)..

John Object’s Structure

It is the principal John Object Daemon configuration. It defines how John Services can see the current John Object.

Object’s Structure contains all Statuses/Actions the Object can provide/execute. With the aim to help Developers identify the right Object to use, all Statuses/Actions are organized into a hierarchy of containers. The main container, common on all John Objects, is the ‘root’ container.

The Object’s Structure depends on John Object’s model. That means all Objects with the same model, share the same Object’s Structure.

Status

Read-only data from John Object, like a Sensor value.

Statuses are the basic components of a John Object’s Structure. They define how the John Object Daemon should translate the John O.S. Protocol messages to/from the Utils & Apps.

Each time a Status updated his value, a ‘Status Update’ message is sent to all John Services connected.

Actions

John Object executable commands from a John Service.

Like Statuses, Actions are the basic components of a John Object’s Structure, but in addition to Statuses, Actions allow John Services execute a specific command on the John Object.

Actions extends Statuses. Then, Actions also share a value to the John EcoSystem and send ‘Status Update’ message on value change.

Service​

John Service

The Service of a John EcoSystem.

Any software can be a John Service, just include the John Service Library.

Once the JSL is included, any software can list, query and interact with all available John Objects.

The JSL manages Direct and Cloud Communication with John Objects.

The JSL also provides user authentication methods. So softwares can ask End User to login with their JCP Accounts and then access to restricted John Objects.

John Manager Service

Like other John Services but it can also access Makers and Developers info.

Makers and Developers can obtain special John Cloud Platform accounts that allow them to manage released John Objects or get info on how End Users interact with their John Services.

John Admin Service

Like John Manager Service but for John Cloud Platform admins.
